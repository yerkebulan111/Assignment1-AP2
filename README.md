# AP2 Assignment 3 — Event-Driven Architecture with RabbitMQ

## Architecture Overview

```
POST /orders
     │
     ▼
Order Service ──gRPC──► Payment Service ──publish──► RabbitMQ (payment.completed)
                                                              │
                                                              ▼
                                                    Notification Service
                                                    (logs email to console)
```

**Flow:**
1. Client sends `POST /orders` to **Order Service**.
2. Order Service calls **Payment Service** via gRPC (`ProcessPayment`).
3. Payment Service persists the payment to PostgreSQL.
4. **After DB commit**, Payment Service publishes a `PaymentEvent` JSON message to the `payment.completed` RabbitMQ queue.
5. **Notification Service** consumes the message, checks idempotency, logs the notification, then sends a **manual ACK**.

## Idempotency Strategy

**Problem:** RabbitMQ guarantees *at-least-once* delivery. A message may be redelivered if the consumer crashes before ACKing.

**Solution:** Each `PaymentEvent` carries a unique `event_id` (UUID generated by the publisher). The Notification Service maintains an `IdempotencyStore` that records every processed `event_id`.

```
On receive message:
  1. Unmarshal JSON → PaymentEvent
  2. Check IdempotencyStore.IsProcessed(event.EventID)
     → YES: ACK immediately, skip logging (no duplicate log)
     → NO:  Log notification, MarkProcessed(event.EventID), then ACK
```

**Current implementation:** In-memory `sync.Map`-backed store (fast, zero dependencies).  
**Production upgrade:** Replace with a Redis `SETNX` call or a `processed_events` PostgreSQL table for persistence across restarts.

## ACK Logic

| Scenario | Action |
|---|---|
| Message processed successfully | `msg.Ack(false)` — removed from queue |
| Duplicate (already processed) | `msg.Ack(false)` — removed, no re-log |
| Unmarshal / permanent error | `msg.Reject(false)` — sent to DLQ via DLX |
| Transient error, retries < 3 | `msg.Nack(false, true)` — requeued |
| Transient error, retries ≥ 3 | `msg.Reject(false)` — sent to DLQ |

Auto-ACK is **disabled** (`autoAck: false` in `channel.Consume`). A message is only acknowledged after the notification log is successfully printed, guaranteeing at-least-once delivery.

## Reliability Guarantees

| Feature | Implementation |
|---|---|
| **Durable queue** | `QueueDeclare(..., durable=true, ...)` — queue survives broker restart |
| **Persistent messages** | `DeliveryMode: amqp.Persistent` — messages written to disk |
| **Manual ACKs** | `autoAck=false`; ACK only after successful processing |
| **At-least-once delivery** | NACK + requeue on transient failures |
| **Dead Letter Queue (DLQ)** | `payment.dlx` exchange + `payment.completed.dlq` queue; messages land here after 3 failed retries or permanent errors |
| **Graceful Shutdown** | All services listen for `SIGINT`/`SIGTERM` via `os/signal` and drain in-flight work before exiting |
| **Startup retry** | Both publisher and consumer retry RabbitMQ connection up to 10 times with 3s backoff |

## Bonus: Dead Letter Queue (DLQ)

The main queue `payment.completed` is declared with:
```json
{
  "x-dead-letter-exchange": "payment.dlx",
  "x-dead-letter-routing-key": "payment.completed.dlq"
}
```

When a message is rejected without requeue (`Reject(false)` or `Nack(false, false)`), RabbitMQ automatically routes it to `payment.completed.dlq`.

**To simulate DLQ movement:** Send a message with an invalid JSON body. The consumer will fail to unmarshal it and immediately reject it → visible in the DLQ via RabbitMQ Management UI.
